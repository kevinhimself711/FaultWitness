# 需求证据治理方法

## 目的

本目录把岗位 JD、真实面经、官方技术材料和上游立项报告转换为可追踪、可验证、可被后续 Gate 消费的需求资产。它不把出现频率等同于正确性，也不把上游总结重复包装成一手证据。

## 语料边界

- 22 份非空岗位 JD，作为 Tier A 的公开招聘需求证据。
- 84 份非空面试材料：牛客渠道 45 份，小红书渠道 39 份；其中一个合集只作为 Tier C 索引入口。
- 2 份空面试材料，保留摘要与排除原因，但不参与需求论证。
- 1 份官方技术参考，作为 Tier A 的协议或实现语义证据。
- 1 份上游立项报告，作为 Tier C 的派生结论与覆盖检查，不作为强制需求的唯一依据。

原始文件不进入仓库。`SOURCE_CATALOG.yaml` 只保存匿名 ID、内容摘要、字节数、字符数、来源类别、证据等级和主题标签，用于证明去重、覆盖和变更检测，不传播第三方原文或私人痕迹。

## 证据等级

| 等级 | 定义 | 可支持的结论 |
| --- | --- | --- |
| Tier A | 岗位 JD 或官方技术材料 | 岗位能力、职责、协议及平台约束 |
| Tier B | 单份候选人真实面经 | 面试追问深度、失败模式、项目解释要求 |
| Tier C | 合集或上游派生报告 | 导航、归纳、交叉检查，不得单独支撑 mandatory 需求 |

强制需求必须至少引用一项 Tier A 或 Tier B 证据。被排除的材料不得被需求或证据矩阵引用。

## 转换流程

1. 对全部源材料做文件级完整性、空文件和内容摘要检查。
2. 以来源内明确出现的主题进行多标签索引；标签只用于检索，不直接决定优先级。
3. 将上游报告的 57 项正式要求逐项回到 Tier A/B 材料做交叉支撑。
4. 把需求落到目标岗位、五个架构平面和 G00-G10 验证 Gate。
5. 为每项需求写出可证伪的验证方式，避免“支持”“完善”“企业级”等不可检验表述。
6. 用 `EVIDENCE_MATRIX.yaml` 检查 57 项需求恰好覆盖一次，并保留跨类别综合论证。

## 纳入和优先级规则

- P0：主链路正确性、安全性、可靠性、可复现性或面试深度的必要条件，标记为 `mandatory: true`。
- P1：必须纳入总体架构与开发蓝图，但允许在主链路稳定后实施，标记为 `mandatory: false`。
- P2：算法训练、多租户 Agent Runtime 和多模态仍是整体项目范围，不是独立附属项目；它们必须满足各自进入 Gate 的前置证据。
- 频繁出现但无法形成可测试行为的材料只保留为主题索引，不制造需求。
- 单个岗位或单篇面经的特殊技术名词，不在缺少项目价值和验证路径时升级为强制范围。

## 自动约束

`make verify-fast` 会拒绝来源计数漂移、重复 ID、未知或已排除的引用、Tier C 单独支撑 mandatory 需求，以及证据矩阵的缺失或重复覆盖。任何语料变化都必须形成新的目录版本和对应 Iteration/Eval 证据。
