# AMD-0002 — Preserve Operator-Declared Long-Lived Credentials

Status: accepted

Date: 2026-07-22

Owner: project owner

## Context

The original G01 plan treated the untracked owner-host `envs.txt` handoff as proof that its server password and API keys were already exposed, and therefore required three rotations. The project owner clarified that all three credentials are intentionally long-lived, must keep their existing values, and are to be used throughout the project. Repository and history checks show that the handoff was never committed.

## Amendment

- I-0007 migrates the existing server password, Bailian key, and LangSmith key unchanged into the repository-external SOPS + Age store.
- Private metadata records explicit operator acceptance; it stores no value, suffix, reversible hash, host locator, or login identity.
- The existing server password must pass a real login while installing the dedicated SSH key.
- Bailian and LangSmith live validity remain owned by I-0013 and I-0014 because I-0007 forbids model and LangSmith API calls.
- `envs.txt` is deleted and precisely ignored after acceptance, server login, host pin, and dedicated-key login; repeated sanitized capability evidence then binds the resulting implementation candidate SHA and remains mandatory for Eval pass.
- Confirmed compromise, a publication/history scan hit, or operator revocation triggers rotation; otherwise the accepted values remain unchanged.

## Impact

This amendment removes unnecessary replacement workflows and corrects the exposure premise. It does not change the public-repository Secret count of zero, host-pin requirement, dedicated-key requirement, SOPS round-trip requirement, capability floor, model/trace Eval ownership, or any product/runtime boundary. No stored business data or runtime migration exists yet.
