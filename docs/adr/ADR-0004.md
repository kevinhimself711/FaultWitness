# ADR-0004: Store Large Artifacts and Datasets Locally With MinIO and DVC

Status: accepted

Date: 2026-07-22

## Context

Evidence bundles, extracted documents, trajectories, datasets, checkpoints, and model artifacts can exceed database, event, and Agent-context limits. They also need content identity, lineage, deletion, rollback, and reproducibility.

## Decision

Use MinIO as the private object store and DVC metadata for versioned dataset/model lineage. PostgreSQL, Events, Trace, and Agent State carry typed artifact references, hashes, bounded summaries, tenant/data classification, and version IDs rather than unbounded payloads.

Eval/Data Worker owns Artifact/Dataset state. Locked evaluation and training artifacts use separate identities and prefixes; Agent Runtime cannot read locked Ground Truth.

## Alternatives

- Store blobs in PostgreSQL: rejected because it couples operational state to large artifact lifecycle and backup cost.
- Commit artifacts to Git: rejected for size, privacy, license, and history-removal reasons.
- Cloud-specific managed storage: rejected as the baseline because local reproducibility is a project requirement.

## Consequences

MinIO/DVC availability, integrity checks, garbage collection, and access policy require explicit operations. MinIO outage pauses large-result flows; the system cannot inject unlimited data into context as a fallback. Artifact digests allow replay and tamper detection.

## Migration and rollback

Storage adapters preserve artifact IDs, content hashes, tenant scope, and lineage. Migration requires copy verification and dual-read before cutover. Rollback restores the previous adapter and catalog mapping without changing referenced content identity.
