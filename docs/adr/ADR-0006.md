# ADR-0006: Make Action Executor the Sole Privileged Write Boundary

Status: accepted  
Date: 2026-07-22

## Context

Agent outputs, retrieved documents, tool results, and generated code are untrusted. Allowing Agent nodes, general tools, MCP integrations, or Sandbox jobs to write the SUT would create approval bypasses and make compensation, idempotency, and audit inconsistent.

## Decision

Action Executor is the only component with SUT write credentials. Agent Worker may emit only an `ActionProposal`. The Executor accepts versioned typed action adapters and revalidates tenant, environment, policy, risk, immutable digest, approval, resource preconditions, idempotency, and expiry before dispatch.

R2 requires a matching immutable ApprovalGrant. Any changed parameter, target version, environment, or scope invalidates the grant. Successful commit requires verified postconditions. Unknown effects enter `UNCERTAIN`; failed postconditions compensate or enter manual escalation. Arbitrary shell and Kubernetes write operations are prohibited.

## Alternatives

- Direct tool execution by Agent: rejected because prompt injection can become privileged execution.
- Approve free-form text: rejected because approval meaning can drift between proposal and dispatch.
- Let Sandbox perform actions: rejected because compute isolation is not an authorization or transaction boundary.

## Consequences

Action adapters require more explicit engineering and contract tests. The design sharply limits blast radius and yields one auditable enforcement point. R1 may skip human approval only through versioned policy; it never skips Executor validation.

## Migration and rollback

New action types require versioned schemas, policy, dry-run, verification, compensation, and Eval. Replacing the Executor requires equivalent digest and transaction semantics with dual-run evidence. Rollback disables the new adapter or service and leaves unresolved transactions for reconciliation, never direct retry.
