# ADR-0005: Adopt At-Least-Once Delivery With Idempotency and Fencing

Status: accepted

Date: 2026-07-22

## Context

Workers, publishers, networks, and dependencies can crash between side effect and acknowledgement. Claiming distributed exactly-once would hide unavoidable uncertainty, especially for external actions that do not share a transaction with FaultWitness.

## Decision

Runtime Tasks and domain Events use at-least-once delivery. Each attempt has a new `attempt_id` under a stable `task_id`. Consumers deduplicate `event_id` and business idempotency keys. Scheduler issues monotonic fencing tokens, and every checkpoint, completion, or dispatch from a Worker verifies the current token.

External write actions have separate idempotency keys, resource preconditions, receipts, postcondition verification, and an explicit `UNCERTAIN` state. A lost response is reconciled before any retry.

## Alternatives

- Exactly-once claim: rejected because external systems and crash windows make it false.
- At-most-once delivery: rejected because lost scheduling or evidence work would be silent.
- Retry without fencing: rejected because stale Workers could overwrite checkpoints or dispatch duplicate work.

## Consequences

Duplicate delivery is normal and tested. Storage must retain dedupe records for a defined window, and operations must monitor dead letters, lease churn, and reconciliation backlog. Correctness depends on typed terminal reasons instead of hiding uncertainty.

## Migration and rollback

Queue or Scheduler implementations may change if stable task/event IDs and fencing semantics remain. Migration requires replay and stale-worker tests. Rollback drains new leases, restores the prior scheduler, and preserves monotonic token history.
