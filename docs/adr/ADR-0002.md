# ADR-0002: Coordinate State Owners With Transactional Outbox Commands and Events

Status: accepted  
Date: 2026-07-22

## Context

Incident, Runtime Task, Agent Graph, and ActionTransaction have different owners and recovery semantics. Direct database updates across services would make partial failures, replay, version conflicts, and audit reconstruction ambiguous.

## Decision

Each owner writes only its state. A state change and its outbound event are committed atomically through a transactional outbox. Cross-owner intent is a typed Command; observed results are typed Events with schema version, tenant, correlation, causation, monotonic sequence, and deduplication identity.

Consumers are idempotent. State mutations require the current `state_version`; Runtime Task commits additionally require a valid fencing token. Terminal state is immutable and later information appends as events.

## Alternatives

- Shared database tables and cross-service writes: rejected because ownership and partial-failure behavior become untestable.
- Distributed two-phase commit: rejected because external tools/actions and most dependencies cannot participate reliably.
- Best-effort publish after state commit: rejected because it can lose the only scheduling or audit notification.

## Consequences

Publication may repeat and consumers must deduplicate. Schema evolution and replay compatibility become first-class. Operational lag is visible as outbox backlog rather than silently missing events. Commands never grant authorization by themselves; the receiving owner revalidates scope and state.

## Migration and rollback

A future transport can replace the publisher without changing Command/Event semantics. A superseding ADR must define dual-read/write, replay, event-version compatibility, failure recovery, and rollback. G00 has no production data migration.
