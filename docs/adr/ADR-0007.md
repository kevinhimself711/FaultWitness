# ADR-0007: Deploy Single-Node K3s Alongside Existing Docker Without Public Ingress

Status: accepted

Date: 2026-07-22

## Context

The target Ubuntu server already runs Docker workloads on ports 80 and 443 and has sufficient CPU, memory, NVMe storage, KVM, and an RTX 4090. G01 needs a real Kubernetes, data-service, isolated-runtime, accelerator, and observability foundation without displacing those workloads or pretending that one host provides high availability.

## Decision

Deploy a pinned single-node K3s cluster on the host alongside Docker. Disable Traefik and ServiceLB, use non-overlapping pod and service CIDRs, expose application services only as `ClusterIP`, and access management endpoints through a controlled SSH tunnel. Establish namespace quotas, default-deny NetworkPolicies, encrypted Kubernetes Secrets, embedded-etcd snapshots, and resource reservations that leave capacity for existing Docker workloads.

Before and after bootstrap, record a sanitized inventory of Docker containers, ports, health, networks, GPU, storage, and host capabilities. Any existing-workload stop, recreation, port conflict, health regression, or unexpected public listener fails the Iteration. K3s version selection must remain compatible with the host kernel and cgroup mode; incompatibility stops implementation and requires a new OS or VM ADR.

## Alternatives

- Replace Docker workloads with Kubernetes: rejected because those services are outside project scope.
- Run K3s in a nested VM: not selected because the owner chose host coexistence and the current host exposes the required KVM and GPU capabilities directly.
- Use Docker Compose only: rejected because it cannot prove the Kubernetes scheduling, policy, RuntimeClass, and recovery behavior required by later Gates.

## Consequences

The topology is realistic for one-node development and evaluation but provides no node-level high availability. Host resource contention and network collisions become explicit failure modes. G01 must preserve rollback instructions and exact pre-bootstrap evidence.

## Migration and rollback

Bootstrap is allowed only in I-0008 after I-0007 secures credentials and captures the baseline. Rollback removes only project-owned K3s services, files, routes, and firewall entries after resolving their absolute targets, then verifies the original Docker inventory and health. It never deletes or recreates pre-existing Docker resources.
